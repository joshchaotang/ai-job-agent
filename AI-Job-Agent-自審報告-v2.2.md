### 一、2025-2026 求職市場核心痛點（數據驅動）

根據多項調查研究，以下是求職者最深刻的痛點：

| 排名 | 痛點 | 數據佐證 | 與本專案的關聯 |
|------|------|----------|--------------|
| 1 | 石沉大海（Ghosting） | 90% 求職者被已讀不回，50% 曾被雇主完全無視 | 可追蹤申請狀態 |
| 2 | 投遞黑洞 | 28% 認為只有 11-25% 的履歷被人看到，44% 過去一個月零面試 | AI 匹配可提高命中率 |
| 3 | 時間拉長 | 中位數從 57 天延長到 83 天，25% 求職者已找超過一年 | 自動化可節省時間 |
| 4 | ATS 機器篩選 | 98.4% 的 Fortune 500 使用 ATS，40% 履歷未經人眼就被淘汰 | 可加入 ATS 相容性檢查 |
| 5 | 心理健康 | 72% 表示求職影響心理健康，79% 感到焦慮 | UI 需要正向回饋 |

| 7 | 技能落差 | 37% 的人技能與職缺不匹配 | AI 分析已涵蓋此功能 |
| 8 | 客製履歷疲勞 | 26% 表示為每份工作客製履歷太耗時 | 可自動產生客製版本 |

### 三、AI Job Agent 功能優化路線圖

基於痛點分析和競品對比，以下是建議的功能增強方向：

#### Phase 1（畢業論文可展示 — 低開發成本）

（1）申請狀態追蹤器
- 痛點對應：石沉大海 + 投遞黑洞
- 實作方式：在每張卡片加入「已儲存 → 已投遞 → 面試中 → 結果」狀態按鈕
- 資料儲存：localStorage（免後端）
- 展示價值：解決 90% 求職者最大痛點

（2）AI 面試問題生成器
- 痛點對應：面試準備
- 實作方式：點擊「面試準備」按鈕 → 用 DeepSeek 根據職缺描述生成 5-8 個可能面試題目
- API 成本：每次約 0.001 USD
- 展示價值：差異化功能，大多數免費工具沒有

（3）技能差距視覺化
- 痛點對應：技能落差
- 實作方式：雷達圖（已有長條圖基礎，升級為 Canvas 雷達圖）
- 展示價值：直覺看出自己的強項和弱項

（4）職缺收藏夾
- 痛點對應：管理混亂
- 實作方式：localStorage 儲存收藏職缺，新增「我的收藏」頁籤
- 展示價值：實用功能，提升使用者黏著度

#### Phase 2（進階功能 — 中開發成本）

（5）一鍵客製求職信
- 痛點對應：客製履歷疲勞
- 實作方式：結合履歷 + 職缺描述 → AI 生成客製求職信
- 展示價值：省下 26% 求職者最頭痛的時間

（6）薪資市場洞察
- 痛點對應：資訊不透明
- 實作方式：整合免費薪資 API，顯示該職位的市場薪資範圍
- 展示價值：幫助求職者談薪水

（7）電子郵件報告
- 痛點對應：效率
- 實作方式：已有 Gmail 設定欄位，加入定時/手動發送搜尋報告功能
- 展示價值：被動接收匹配職缺

#### Phase 3（論文加分項 — 高展示價值）

（8）多平台職缺聚合
- 增加更多免費 API：LinkedIn Jobs（非官方）、GitHub Jobs、HN Who's Hiring
- 展示價值：資料量越大，AI 分析越有意義

（9）ATS 相容性檢查
- 用 AI 分析使用者的履歷，給出 ATS 友善度評分
- 展示價值：直接解決 40% 履歷被機器淘汰的痛點

（10）求職數據儀表板
- 視覺化統計：投遞數、面試率、回應率、各平台表現
- 展示價值：讓使用者看到自己的求職「健康度」

### 四、建議的畢業論文展示優先級

如果時間有限，建議聚焦以下三個功能，論文展示效果最佳：

1. 申請狀態追蹤器（解決最大痛點 + 數據支持）
2. AI 面試問題生成器（差異化 + 技術含量高）
3. 技能差距雷達圖（視覺衝擊力強）

## 第二部分：開發流程自審（五行五臟閉環）

### 一、流程回顧（水・根基 → 腎・記憶官）

整個開發過程的時間線：

| 階段 | 做了什麼 | 結果 |
|------|----------|------|
| 前期（上一個對話） | 嘗試 SunRise AI 全新 UI 設計 | 被使用者否決，教訓：先問再做 |
| 還原 v1.4 | 還原深色主題 + 卡片佈局 + 設定面板 | 成功 |
| 雲端化 | 加入 Remotive + Arbeitnow 搜尋 API | 成功 |
| API Key 架構 | 前端 localStorage + 後端接收 | 成功 |
| 部署 v2.1 | 修復 Arbeitnow 型別錯誤 → 部署 | 成功但有遺留問題 |
| 自審 + v2.2 | 識別 3 個問題 → 全部修復 | 等待最終部署 |

### 二、做得好的地方（土・結構 → 心・審查員）

（1）架構設計正確
- 使用者自帶 API Key 的安全架構（localStorage → 前端傳送 → 伺服器不儲存）
- HMAC-SHA256 Token 驗證機制
- Vercel Serverless Functions 無伺服器架構

（2）防禦性程式設計
- str() 包裝所有外部資料
- isinstance 型別檢查
- try/except 隔離每個 API 來源

（3）使用者體驗
- 深色主題 + 卡片佈局 + 分數圓圈 + 長條圖
- 設定面板一目了然（API Key、模型選擇、履歷、Gmail）
- CSV 匯出功能

### 三、需要改進的地方（木・規劃 → 肝・架構師）

（1）「先做再問」的教訓
- 問題：SunRise AI 重新設計沒有事先確認就做了
- 根因：假設使用者想要「更好的設計」，沒有確認「更好」的定義
- 改善：重大 UI 變更必須先給 mockup 或文字描述讓使用者選擇
- FM 關聯：無（這是流程問題，不是技術 bug）

（2）驗證閘門不夠嚴
- 問題：修復 created_at 時只驗了「不崩潰」，沒驗「顯示值合理性」
- 根因：把「修復 = 不報錯」當成完成條件，忽略了「輸出值是否符合預期」
- 改善：每次修復後，驗證清單必須包含「輸出值是否對人類有意義」
- FM 關聯：FM-001

（3）靜默錯誤處理
- 問題：Remotive 失敗被 catch 吞掉，使用者只看到 2.5 秒的 toast
- 根因：把「不讓使用者看到錯誤」當成好的 UX，其實是隱藏了重要資訊
- 改善：錯誤必須有持久的 UI 呈現（紅色警告條），不能只靠 toast
- FM 關聯：FM-002

（4）部署環境限制未提前識別
- 問題：直到要部署才發現 VM 網路封鎖 npm + Vercel API
- 根因：沒有在流程早期測試部署路徑
- 改善：接到部署任務時，第一步應該是測試部署管道是否通暢
- FM 關聯：FM-003

（5）邊界值測試覆蓋不足
- 問題：Unix 時間戳 1770741231 是 10 位數，str()[:10] 不會截斷
- 根因：測試只用了「壞掉的場景」，沒用「邊界值場景」
- 改善：每次修改字串處理邏輯時，必須用這些值測試：空值、整數、超長字串、特殊字元

### 四、失敗模式登記冊（金・呈現 → 肺・品質官）

| FM 編號 | 描述 | 觸發場景 | 防護措施 |
|---------|------|----------|----------|
| FM-001 | 型別轉換通過但顯示不合理 | API 回傳整數值，str() 後長度恰好不超過切片範圍 | 除了型別轉換，還要做格式驗證 |
| FM-002 | try/except 靜默吞錯導致資料缺失 | 多來源搜尋中某一來源失敗 | 失敗來源必須在 UI 持久顯示 |
| FM-003 | 部署環境網路限制 | Cowork VM 封鎖外部服務 | 接到部署任務時先測試管道 |
| FM-004 | 未確認就做重大 UI 變更 | 使用者期望與開發者假設不符 | 重大變更先給選項確認 |
| FM-005 | 邊界值測試覆蓋不足 | 修復後只驗「不報錯」 | 測試清單必須包含邊界值 + 輸出合理性 |

### 六、大周天框架自身的精進建議

經過這次專案實戰，對大周天框架 v4.6 的改進建議：

（1）驗證閘門增加「輸出合理性」子項
- 現有：功能驗證、邊界驗證、錯誤驗證、視覺驗證、回溯驗證、清理驗證
- 建議新增：「語意驗證」— 輸出的值對人類來說是否有意義？
  例如：日期欄位顯示 "1770741231" 在技術上不是錯誤，但對人類沒意義

（2）強制追問清單增加一項
- 現有 6 項（資料缺漏、快取損壞、篩選後數量、操作時機、跨環境差異、假設差異）
- 建議新增：「轉換後的值是否仍然保有原始語意？」
  例如：整數 → 字串後，值的「意義」是否改變了？

（3）部署前增加「管道測試」步驟
- 在火・執行之前，增加一個「部署管道可行性」的前置檢查
- 確認：能連到部署服務嗎？有權限嗎？網路通嗎？
- 避免做完所有開發後才發現無法部署
